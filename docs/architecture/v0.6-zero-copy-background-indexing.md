# ColQL v0.6.0 Architecture Lock: Zero-Copy Background Indexing

## 1. Purpose

This document is the architectural source of truth for ColQL v0.6.0 before implementation begins. It locks the memory model, worker model, index lifecycle model, query fallback behavior, and correctness invariants for Zero-Copy Background Indexing.

The goal is to avoid repeatedly revisiting foundational decisions before v1.0.0. Implementation details may still be refined by tests and benchmark evidence, but the constraints in this document are not optional.

ColQL is a TypeScript-first, zero-runtime-dependency, process-local, in-memory columnar query engine. The v0.6.0 architecture must preserve that positioning.

## 2. Non-goals

v0.6.0 must not include:

- Manual rebuild APIs as the primary user-facing solution.
- Full table or full column copies to workers for large rebuild input.
- One worker per chunk.
- Object row snapshots for worker input.
- Incremental sorted index rebuild.
- Compound indexes.
- Plan cache.
- Persistence.
- Distributed coordination.
- SQL.
- Runtime dependencies.
- Browser worker support unless it works naturally from the Node-first architecture without extra API promises.

Manual rebuild methods may remain for compatibility if they already exist, but they must not become the product answer to dirty-index latency.

## 3. Current Problem

v0.5.0 hardened ColQL around trust and stability: type-test gate, column-scoped index invalidation, hardened serialization, numeric wire-format versioning, deterministic mutation/index/serialization fuzzing, additive query diagnostics, CodSpeed coverage, and clearer process-local/non-durable/non-SQL documentation.

The remaining problem is index rebuild lifecycle.

A 10M-row session analytics benchmark showed strong hot-path indexed performance:

- Equality index query on 10M rows: about 10.8ms.
- Sorted range query on 10M rows: about 16.6ms.
- Callback full scan: about 2218ms.
- First query after dirty sorted index: about 6126ms.
- Second query after sorted rebuild: about 424ms.
- Recreate explicit indexes after restore: about 14323ms.

The important spike path is the dirty sorted-index rebuild. Indexed hot paths are already strong enough to validate the indexing strategy, but a large dirty sorted index can still force the first user-facing query after mutation to synchronously pay a multi-second rebuild cost.

v0.6.0 moves large rebuild work off the user-facing query path while preserving correctness.

## 4. Final v0.6.0 Product Decision

Default effective background indexing mode is locked as auto. The public configuration field is `indexing.background`:

```ts
indexing: {
  background: "auto",
}
```

Default behavior:

- Background indexing is enabled by default in conservative auto mode.
- Small tables may still rebuild synchronously.
- Large dirty index rebuilds should use background indexing when memory-safe.
- The query path should fall back rather than synchronously pay a huge rebuild cost.
- If background indexing cannot run safely, ColQL must use correctness-preserving fallback behavior.
- If a background rebuild would exceed a configured or estimated memory budget, ColQL must not start the background job. It must preserve correctness through fallback behavior instead of creating large temporary copies or unbounded rebuild memory.

Correctness always wins over latency. Queries must never return stale index results.

`indexing.background: false` must preserve the v0.5 synchronous dirty-index lifecycle behavior as a rollback and safety mode. This mode must remain available throughout the v0.6.0 implementation phases.

Manual rebuild or clean APIs are not part of the primary public API for v0.6.0. Users should define indexes; ColQL should manage index lifecycle internally.

Users should not normally need to call APIs such as:

- `rebuildDirtyIndexes()`
- `cleanIndexes()`
- `forceRebuild()`
- `prewarmIndexes()`

## 5. Storage Model Decision

Current storage summary:

- Numeric columns use typed-array chunks.
- Dictionary columns store integer codes in typed-array chunks.
- Boolean columns use bit-packed byte chunks.
- The default chunk size is currently 65,536 logical rows.
- Current chunks are normal `ArrayBuffer`-backed typed arrays, not `SharedArrayBuffer`-backed typed arrays.

Target storage model:

- Worker-readable chunks require `SharedArrayBuffer`-compatible storage.
- The migration must replace storage backing; it must not add duplicate steady-state storage beside existing buffers.
- Numeric columns and dictionary code columns are the first background-indexing target.
- Boolean chunks may be deferred unless current public indexing requires them.
- Dictionary string values must not be sent to workers for index rebuild; workers operate on dictionary codes.

Locked decisions:

- `SharedArrayBuffer` is the architectural dependency for real large-table worker indexing.
- Transferable live input buffers are rejected because transferring detaches table storage.
- Copying chunks is rejected for large rebuild input.
- Copy-on-write snapshots are deferred unless they become necessary for correctness or memory safety.
- Background worker indexing must not materially increase steady-state table memory.
- Temporary rebuild memory must be bounded, measured in benchmarks, observable through diagnostics, and documented.

## 6. Worker Input Memory Model

Workers may receive only compact job and chunk metadata:

- Job id.
- Index id.
- Index kind.
- Column id or name.
- Column type tag.
- Index generation.
- Column epoch.
- Chunk descriptors.
- `SharedArrayBuffer` reference.
- Byte offset.
- Byte length.
- Row start.
- Logical length.
- Chunk size.
- Row count.
- Value or code width.

Workers must never receive:

- Full table objects.
- Row objects.
- Schema object graphs.
- Validation functions.
- Live index objects.
- JS `Map` index state.
- Dictionary string values, except possibly diagnostics-only metadata.
- Copied 10M-column buffers.

Worker input must be zero-copy or near-zero-copy. Any fallback that would copy a full large column to a worker is forbidden.

## 7. Worker Output Memory Model

Worker output must be transferable `ArrayBuffer` data or `SharedArrayBuffer`-backed typed data.

Large JS arrays are forbidden for output.

### Sorted Index Output

Sorted index output uses typed row-id arrays.

`Uint32Array` is acceptable while row count is less than or equal to 4,294,967,295. Larger future row counts require either `BigUint64Array` or a segmented row-id representation.

Ordering rules:

- Sort by column value ascending.
- Break ties by row id ascending.
- Preserve deterministic behavior.
- Invalid numeric values remain rejected by validation before indexing.

The large sorted rebuild path must not allocate a JS `number[]` for all row ids.

### Equality Index Output

Worker equality output must not be JS `Map`-heavy for large rebuilds.

Preferred output is encoded grouped buffers:

- Key buffer.
- Offsets buffer.
- Row ids buffer.

Partial results are merged by key. The merge step may build final lookup structures on the main thread, but the worker boundary must stay typed-buffer based and avoid object-heavy transfer.

The output format must make memory cost observable: number of keys, row-id buffer length, offsets length, and byte length.

### Unique Indexes

Unique indexes stay synchronous and main-thread-only in v0.6.0.

Reason: unique indexes are integrity structures, not just performance structures. They enforce duplicate-key protection and power by-key helpers. Background unique rebuilds would introduce dangerous ambiguity around duplicate detection and mutation ordering.

## 8. Chunk Task Queue and Worker Pool

v0.6.0 must not create one worker per chunk.

Architecture:

```text
column chunks
  -> chunk rebuild tasks
  -> bounded worker pool
  -> partial typed-buffer results
  -> merge
  -> generation/epoch validation
  -> atomic index swap
```

Chunks become tasks. Workers pull tasks from a queue.

Default worker policy:

```ts
workerCount = max(1, min(4, availableParallelism - 1))
```

Use `node:os.availableParallelism()` when available. Fall back to `os.cpus().length` when necessary.

Scheduling:

- FIFO is acceptable for a single rebuild job.
- If multiple rebuild jobs exist, schedule fairly with round-robin across jobs.
- Limit max queued jobs per table.
- Limit active rebuild jobs per table.
- Use generation checks for stale job handling.

Worker lifecycle:

- Workers are created lazily.
- Workers are reused across rebuild jobs.
- Workers must terminate on table dispose or process cleanup when an explicit cleanup surface exists.
- Worker failures mark affected indexes failed and preserve query fallback.

No runtime dependencies are allowed.

## 9. Index Lifecycle State Machine

Stable states:

- `fresh`
- `dirty`
- `queued`
- `rebuilding`
- `failed`

`stale` is a transient result classification, not necessarily a stable public state.

### fresh

Entered after index creation, synchronous rebuild, or successful atomic background swap.

Query behavior: planner may use the index.

Mutation behavior: affected mutations transition the index to `dirty`.

Explain behavior: reports usable fresh index state and candidate count when it can be computed cheaply.

Diagnostics behavior: reports last rebuild mode, duration, generation, and no active job.

### dirty

Entered after affected mutation, restore/recreate dirtying, worker failure retry, or config change.

Query behavior: must not use the dirty index. Small tables may rebuild synchronously. Large tables schedule background rebuild when safe and fall back.

Mutation behavior: increments epoch/generation again as needed and remains dirty.

Explain behavior: non-executing; reports dirty state and must not schedule work.

Diagnostics behavior: reports dirty reason and generation.

### queued

Entered when a background rebuild job has been accepted but workers have not started processing it.

Query behavior: fallback scan or use another fresh index.

Mutation behavior: affected mutation makes the queued job stale by generation/epoch mismatch and marks the index dirty again.

Explain behavior: reports queued/rebuilding-style state without scheduling work.

Diagnostics behavior: reports job id, queued chunks, generation, and dirty reason.

### rebuilding

Entered when one or more worker tasks for the job are active.

Query behavior: fallback scan or use another fresh index.

Mutation behavior: affected mutation invalidates the in-flight result by changing generation/epoch.

Explain behavior: reports non-usable rebuilding state.

Diagnostics behavior: reports chunks done/total, worker count, job id, generation, and elapsed time if cheap.

### failed

Entered when worker execution, merge, output validation, or packaging/runtime failure prevents rebuild completion.

Query behavior: fallback scan or use another fresh index.

Mutation behavior: affected mutation may transition to dirty with a new generation.

Explain behavior: reports failed state and fallback reason.

Diagnostics behavior: reports last error code/message and fallback count.

Dirty reason codes:

- `insert`
- `insertMany`
- `update:indexed-column`
- `delete:row-position-shift`
- `restore`
- `worker-failed`
- `config-change`

## 10. Epoch and Generation Model

Locked model:

- Per-column epoch.
- Per-index generation.
- Optional table epoch for broad operations and diagnostics.

Rules:

- Any affected mutation increments the column epoch.
- Any affected mutation increments the index generation.
- A worker job captures both index generation and column epoch at scheduling time.
- A worker result is accepted only if both generation and epoch still match.
- Stale worker results are discarded.
- Accepting a stale worker result is a release blocker.

Mutation during rebuild:

- Mutation marks the index dirty again.
- The current worker result becomes stale.
- A new job may be scheduled or debounced.
- Query correctness is maintained by fallback scan or another fresh index.

The table may expose table epoch in diagnostics, but table epoch alone is not sufficient to accept a worker result.

## 11. Atomic Index Swap

Background rebuild completion must follow this sequence:

1. Receive worker or merge result.
2. Validate job id.
3. Validate index id.
4. Validate index kind.
5. Validate column id/name.
6. Validate generation.
7. Validate column epoch.
8. Validate output shape and buffer lengths.
9. Build or finalize the new index representation.
10. Atomically replace the old dirty index with the fresh index.
11. Update state and diagnostics.
12. Discard stale or malformed results.

The old dirty index must never be used while rebuilding. The swap point is the only point at which the new index becomes visible to query planning.

## 12. Query Behavior While Rebuilding

Planner behavior is locked:

- Fresh selected index: use it.
- Dirty index on a small table: synchronous rebuild may happen.
- Dirty index on a large table: schedule background rebuild and fallback.
- Queued or rebuilding index: fallback scan or use another fresh index.
- Failed index: fallback scan or use another fresh index.

`explain()` requirements:

- Must remain non-executing.
- Must not scan rows.
- Must not materialize rows.
- Must not call query hooks.
- Must not schedule background jobs.
- Must not rebuild dirty indexes.

Terminal execution may schedule a background rebuild when auto policy allows it.

`onQuery` should report fallback and rebuild state when applicable.

Queries must never return stale index results.

## 13. Background Auto Policy

Initial default thresholds:

```ts
maxSyncRebuildRows = 100_000
backgroundThresholdRows = 100_000
```

These thresholds are starting points and may be tuned by benchmark evidence.

`indexing.background: "auto"` behavior:

- Small tables: synchronous rebuild is allowed.
- Large tables plus zero-copy-capable index: background rebuild.
- Unsafe memory path: fallback, no huge copy and no unbounded temporary rebuild memory.
- Configured or estimated memory budget exceeded: do not start the background job; preserve correctness through fallback behavior.
- Worker failure: fallback.

Public configuration may include:

```ts
{
  indexing: {
    background: "auto" | true | false,
    workerCount?: number,
    maxSyncRebuildRows?: number,
    backgroundThresholdRows?: number,
  },
}
```

Keep the public API minimal. Prefer one table-level indexing config over multiple lifecycle control methods.

`indexing.background: false` preserves the v0.5 synchronous dirty-index lifecycle behavior. It is the required rollback and safety mode while v0.6.0 background indexing is implemented and hardened.

## 14. Diagnostics

Table-level diagnostics should report:

- Index kind.
- Column.
- State.
- Dirty reason.
- Generation.
- Column epoch.
- Queued/rebuilding status.
- Job id.
- Chunks done and chunks total.
- Worker count.
- Last rebuild duration.
- Last rebuild mode.
- Stale results discarded.
- Fallback count.
- Last error code/message.

Explain additions should include:

- `indexState`
- Background indexing effective mode/state derived from `indexing.background`
- `fallbackReason`
- `selectedIndex`

`onQuery` additions should include:

- `backgroundRebuildScheduled`
- `backgroundRebuildState`
- `fallbackReason`
- `indexState`
- `selectedIndex`

Diagnostics must be cheap. They must avoid high-cardinality per-chunk details by default.

## 15. Serialization and Restore

Serialization behavior is locked:

- Serialization does not wait for worker jobs.
- Serialization writes only table data.
- Serialized indexes remain unsupported.
- Worker jobs are never serialized.
- Index state is not serialized.
- Restored tables have no trusted indexes unless indexes are recreated.
- Background indexing runtime config is not serialized unless explicitly decided in a later architecture document.

Serializing while a rebuild is queued or running must produce a valid table snapshot independent of the rebuild.

## 16. Worker Packaging and Runtime

Worker runtime target:

- Node `worker_threads`.
- Node built-ins only.
- No runtime dependencies.

Packaging concerns:

- ESM and CJS output must both locate worker code correctly.
- `tsup` output must include worker files.
- The npm tarball must include worker runtime artifacts.
- Worker file path resolution must work from installed package output, not only from the repository.
- Tests must verify build output, package contents, and worker startup from built artifacts.

Node support:

- Use `node:os.availableParallelism()` when available.
- Fall back to `os.cpus().length`.
- The minimum Node version must be confirmed before release.

Browser support is a non-goal unless the architecture works naturally without new public promises.

Worker packaging is a major release risk for v0.6.0.

## 19. Acceptance Criteria

v0.6.0 must not release unless:

- No full column/table copy is used for large worker input.
- No one-worker-per-chunk design exists.
- Stale worker results are discarded.
- Mutation during rebuild is correct.
- Queries never return stale index results.
- Query can fallback while rebuild runs.
- The 10M dirty sorted first-query rebuild spike is not paid synchronously on the query path.
- Worker output uses transferable/shared typed buffers.
- Peak memory during rebuild is measured and documented.
- `npm test` passes.
- `npm run test:types` passes.
- `npm run build` passes.
- `npm run bench:codspeed` passes.
- Package dry-run passes.
- Docs clearly explain limitations.

Stale result acceptance blocks release.

## 20. Implementation Phase Order

Implementation order is locked:

1. Storage descriptor interface and feasibility checks.
2. Index lifecycle state machine.
3. Epoch/generation tracking.
4. Query fallback semantics with sync implementation still available.
5. `SharedArrayBuffer`-compatible numeric/dictionary code storage.
6. Worker pool and chunk task queue with mock jobs.
7. Equality background rebuild.
8. Sorted background rebuild.
9. Atomic swap and stale result discard hardening.
10. Diagnostics.
11. Benchmarks.
12. Docs/release hardening.

Every phase must preserve a fallback mode equivalent to v0.5 synchronous behavior.

## 21. Risks and Mitigations

Memory spikes:

- Mitigation: no full large input copies; typed-buffer output; benchmark `rss`, `external`, and `arrayBuffers`; expose diagnostics.

Worker leaks:

- Mitigation: bounded pool, explicit cleanup behavior, worker lifecycle tests.

Stale index correctness bugs:

- Mitigation: per-column epoch, per-index generation, atomic swap validation, stale discard tests, mutation-during-rebuild tests.

Flaky async tests:

- Mitigation: deterministic fake workers, barriers, and mocks; avoid sleep-based assertions.

Worker packaging failure:

- Mitigation: built-artifact startup tests, npm tarball checks, ESM/CJS resolution tests.

ESM/CJS output mismatch:

- Mitigation: verify worker paths for both package entrypoints.

`SharedArrayBuffer` migration bugs:

- Mitigation: storage descriptor tests, serialization regression tests, chunk-boundary tests, memory benchmarks.

Equality output memory blowups:

- Mitigation: encoded grouped buffers, no large worker `Map` output, report key/offset/row-id sizes.

Sorted merge memory blowups:

- Mitigation: typed row-id buffers, bounded merge strategy, no full JS `number[]` output.

Public API creep:

- Mitigation: table-level config and diagnostics only; lifecycle control APIs are not the primary public solution.

Zero-dependency promise:

- Mitigation: Node built-ins only.

Browser ambiguity:

- Mitigation: do not promise browser worker support in v0.6.0.

## 22. Final Locked Decisions

- Default background indexing mode is `auto`.
- Public config uses `indexing.background`, not multiple naming variants.
- `indexing.background: false` preserves v0.5 synchronous dirty-index lifecycle behavior.
- Auto mode must respect memory-budget safety and fallback instead of starting unsafe background jobs.
- No full large input copies to workers.
- Use a bounded worker pool.
- Use chunk tasks, not chunk workers.
- `SharedArrayBuffer`-compatible storage is required before real worker indexing.
- Worker input must be zero-copy or near-zero-copy.
- Worker output must be typed, transferable, or shared.
- Numeric columns and dictionary code columns are the first worker-indexing target.
- Dictionary string values are not sent to workers for rebuild.
- Boolean chunks may be deferred unless public indexing requires them.
- Unique indexes stay synchronous and main-thread-only in v0.6.0.
- Query fallback is required while indexes are dirty, queued, rebuilding, or failed.
- `explain()` remains non-executing and does not schedule rebuilds.
- Terminal query execution may schedule background rebuilds in auto mode.
- Manual rebuild APIs are not the primary public API.
- Serialization does not wait for or serialize worker jobs.
- Worker packaging is a major release risk.
- Stale result acceptance blocks release.
